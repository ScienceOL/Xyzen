# Philosophy

Xyzen is built on a small set of opinionated design principles that prioritize horizontal scalability, operational simplicity, and developer ergonomics. {{ className: 'lead' }}

## Stateless Async Execution

The core architectural decision in Xyzen is the strict separation between **connection management** (FastAPI) and **heavy computation** (Celery workers).

API containers remain completely stateless. When a user sends a message, the API pod publishes a task to Celery and immediately returns. The actual LLM orchestration — tool calls, streaming, multi-step reasoning — happens in a worker process that has no knowledge of which API pod the user is connected to.

This means you can scale API pods and worker pods independently. If you need to handle more concurrent connections, add API replicas. If LLM inference is the bottleneck, add worker replicas. The two layers never need to know about each other.

**Ephemeral state** (streaming channels, pub/sub subscriptions) lives in Redis. **Persistent state** (conversations, messages, agent configs) lives in PostgreSQL. API containers hold nothing.

## No-Foreign-Key Database

Xyzen uses **logical references** (`user_id: str`, `agent_id: str`) instead of SQL foreign key constraints. Relationships are enforced in the service layer, not the database layer.

This is a deliberate tradeoff:

- **Simpler migrations** — Adding or removing relationships doesn't require coordinated schema changes across tables
- **Flexible storage** — Individual tables can be moved to different databases or sharded without breaking referential integrity constraints
- **Faster writes** — No cascading constraint checks on INSERT/UPDATE
- **Explicit logic** — Relationship rules are visible in application code rather than hidden in DB constraints

The cost is that you must handle orphaned references and relationship validation in code. Xyzen's [repository pattern](#repository-pattern) provides consistent CRUD operations that handle this.

## Pub/Sub Bridge

Workers process tasks independently and broadcast results back to the specific API pod via Redis channels (`chat:{connection_id}`). This enables the stateless async model:

1. User sends message → API pod creates Celery task
2. Celery worker picks up task, runs LLM chain
3. Worker publishes streaming events to Redis channel
4. API pod subscribes to channel, streams events to user via SSE

The user's SSE connection is tied to one API pod, but the worker doesn't care which pod it is. Redis acts as the bridge. If an API pod restarts, the user reconnects and picks up events from where they left off using Redis Streams' `Last-Event-ID` replay.

## Repository Pattern

All database access goes through repository classes in `repos/`. Business logic lives in `core/` services that compose repository calls.

```
Component → Core Service → Repository → Database
                ↕
            Redis (cache, pub/sub)
```

This keeps database queries centralized and testable. Repositories handle query construction and data mapping. Services handle business rules, validation, and cross-cutting concerns.

## Component Reuse

Agents in Xyzen are composed from **reusable components** — small subgraphs that handle specific capabilities (tool-calling loops, research phases, planning steps). Components are registered globally and resolved at graph compilation time.

This means you can build a new agent type by composing existing components rather than writing everything from scratch. The Deep Research agent, for example, is composed of four components (clarify, brief, supervisor, report) that could each be reused in other agent types.
