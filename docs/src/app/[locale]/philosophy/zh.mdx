# 设计理念

Xyzen 基于一组精心设计的原则构建，优先考虑水平扩展性、运维简洁性和开发者体验。 {{ className: 'lead' }}

## 无状态异步执行

Xyzen 的核心架构决策是**连接管理**（FastAPI）和**重型计算**（Celery Worker）的严格分离。

API 容器完全无状态。当用户发送消息时，API Pod 将任务发布到 Celery 并立即返回。实际的 LLM 编排——工具调用、流式传输、多步推理——发生在与 API Pod 完全无关的 Worker 进程中。

这意味着你可以独立扩展 API Pod 和 Worker Pod。如果需要处理更多并发连接，增加 API 副本即可。如果 LLM 推理是瓶颈，增加 Worker 副本即可。两个层面互不依赖。

**临时状态**（流式通道、发布/订阅频道）存储在 Redis 中。**持久状态**（对话、消息、智能体配置）存储在 PostgreSQL 中。API 容器不保存任何内容。

## 无外键数据库

Xyzen 使用**逻辑引用**（`user_id: str`、`agent_id: str`）而非 SQL 外键约束。关系在服务层而非数据库层进行维护。

这是一个有意为之的权衡：

- **更简单的迁移** — 增删关系无需跨表协调 Schema 变更
- **灵活的存储** — 单个表可以迁移到不同数据库或分片，不会破坏引用完整性约束
- **更快的写入** — INSERT/UPDATE 时无级联约束检查
- **显式逻辑** — 关系规则在应用代码中清晰可见，而非隐藏在数据库约束中

代价是必须在代码中处理孤立引用和关系验证。Xyzen 的[仓库模式](#repository-pattern)提供了一致的 CRUD 操作来处理这些问题。

## 发布/订阅桥接

Worker 独立处理任务，并通过 Redis 频道（`chat:{connection_id}`）将结果广播回特定的 API Pod。这支撑了无状态异步模型：

1. 用户发送消息 → API Pod 创建 Celery 任务
2. Celery Worker 接收任务，运行 LLM 链
3. Worker 将流式事件发布到 Redis 频道
4. API Pod 订阅频道，通过 SSE 将事件流式传输给用户

用户的 SSE 连接绑定在一个 API Pod 上，但 Worker 不关心是哪个 Pod。Redis 充当了桥梁。如果 API Pod 重启，用户重新连接后通过 Redis Streams 的 `Last-Event-ID` 回放机制从中断处继续。

## 仓库模式

所有数据库访问都通过 `repos/` 中的仓库类进行。业务逻辑在 `core/` 服务中组合仓库调用。

```
组件 → 核心服务 → 仓库 → 数据库
          ↕
      Redis（缓存、发布/订阅）
```

这使得数据库查询集中化且易于测试。仓库处理查询构建和数据映射，服务处理业务规则、校验和横切关注点。

## 组件复用

Xyzen 中的智能体由**可复用组件**组成——处理特定功能的小型子图（工具调用循环、研究阶段、规划步骤）。组件在全局注册，并在图编译时解析。

这意味着你可以通过组合现有组件来构建新的智能体类型，而不是从零开始编写。例如，Deep Research 智能体由四个组件（澄清、简报、监督、报告）组成，每个组件都可以在其他智能体类型中复用。
